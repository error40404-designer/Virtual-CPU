#vCPU computer behaviour and intergration simulator
HTR OS Simulator is a modular operating system simulation written in C++. It is designed to emulate the core components of a computer system, including a CPU, memory systems, storage layers, networking, and a command-line terminal interface.

This project focuses on understanding how low-level systems interact, while maintaining a structured and extensible codebase suitable for further development.

---

## Overview

HTR OS is not a full operating system, but a simulation of one. It models how different hardware and software components communicate, execute instructions, and manage data.

The system is built using multiple independent modules, each representing a core subsystem. These modules are connected through clearly defined interfaces, allowing for scalability and experimentation.

---

## Features

### CPU Simulation
- Executes instructions generated by the assembler
- Interacts with RAM for data storage and retrieval
- Designed to be extensible for future instruction sets

### RAM (Memory Management)
- Stores key-value data
- Supports read and write operations
- Provides memory usage statistics

### Storage Systems
#### SSD
- Fast-access storage simulation
- Used for quick data retrieval

#### HDD
- Persistent storage simulation
- Can be extended for file-based storage systems

### Assembler
- Converts high-level instruction input into executable instructions
- Bridges the gap between user commands and CPU execution

### Terminal Interface
- Command-line interface for interacting with the system
- Supports custom commands such as:
  - File creation (`touch`)
  - System queries
  - Memory access commands

### File Handling (Basic)
- Create files directly from the OS terminal
- Uses real file system via C++ file streams
- Can be extended into a virtual file system

### Network Module
- Placeholder for network-based operations
- Structured for future expansion (e.g., request handling, communication protocols)

## Development Approach

This project was developed incrementally, focusing on one subsystem at a time.

### Phase 1: Core Structure
- Defined basic modules: CPU, RAM, Storage
- Established header and source file separation
- Ensured modular compilation

### Phase 2: Execution Flow
- Connected CPU with RAM
- Introduced instruction handling
- Built assembler to generate instructions

### Phase 3: Storage Integration
- Added SSD and HDD simulation layers
- Designed basic data flow between memory and storage

### Phase 4: Terminal System
- Built a command parser
- Enabled user interaction with system components
- Added file creation capability

### Phase 5: Debugging and Build System
- Resolved linker errors and dependency issues
- Standardized compilation process
- Optimized build using compiler flags

This iterative process ensured that each component was functional before integrating it into the larger system.
